# 서비스 실행 계획

## 목표
AI 협업 보안 코드 생성 기능을 실제 사용자 업무에 투입 가능한 서비스로 운영합니다.

성공 기준은 "그럴듯한 시연"이 아니라 다음 4가지입니다.
- 정확성: 생성 결과의 품질과 일관성
- 안정성: 오류 발생 시 복구 가능하고 재시도 가능한 실행 흐름
- 운영성: 로그, 설정, 인증, 실행 이력 관리가 가능한 구조
- 확장성: provider/역할/실행 모드를 운영 중에도 안전하게 확장 가능

## 팀 역할 분담

### A: Core Logic / AI Engine
서비스 품질을 책임지는 실행 엔진 담당입니다.

주요 책임:
- 멀티 에이전트 프롬프트와 역할 정책 고도화
- 코드 후보 생성과 평가 기준 정교화
- Red Team/Blue Team/Judge 로직의 실제 운영 안정성 확보
- 실패 케이스(타임아웃, provider 실패, 불완전 응답) 처리 정책 수립

핵심 흐름:

```text
요청 입력 -> 후보 생성 -> 보안 검토 -> 합의/선정 -> 결과 + 근거 반환
```

### B: CLI / Runtime / Operations
실제 사용자 실행 경험과 운영 관측 가능성을 책임집니다.

주요 책임:
- CLI 명령 체계 안정화(`run`, `login`, `providers`, `assign` 등)
- 실행 로그/에러 메시지/종료 코드 표준화
- 설정 및 인증 상태 관리(재실행 가능성 보장)
- 운영 관점의 사용성 개선(빠른 실패 원인 파악, 재시도 용이성)

핵심 역할:

```text
터미널 실행 경험 = 서비스 신뢰도
```

## 6시간 실행 타임라인 (서비스 기준)

### 0~1시간: 운영 요구사항 정리
A:
- 역할별 출력 계약(필수 필드, 실패 시 fallback) 정의
- 보안 검토 기준과 점수 체계 정의

B:
- CLI 사용자 플로우 확정(초기 로그인 -> 실행 -> 결과 확인)
- 운영 로그/오류 분류 기준 수립

### 1~3시간: 핵심 기능 구현
A:
- 후보 생성/보안 분석/합의 로직 구현 고도화
- provider 오류와 비정상 응답 처리 추가

B:
- 실행 명령/옵션 안정화
- 오류 메시지와 종료 코드 일관성 확보

### 3~5시간: 안정성 및 운영성 강화
A:
- 결과 품질 검증 시나리오 점검
- 취약점 분석 근거 출력 형식 통일

B:
- 재현 가능한 실행 로그 구조 확정
- 구성 상태 확인 명령(`providers`, `assign`, `use`) 점검

### 5~6시간: 릴리스 전 점검
- 회귀 테스트 실행
- 주요 사용자 시나리오 수동 검증
- 실패 복구 경로(재시도/재로그인/provider 전환) 검증
- 운영 문서/온보딩 가이드 정리

## 서비스 MVP 범위

### 반드시 포함할 기능
- 코드 후보 생성 및 근거 기반 최종 선택
- Red/Blue/Judge 분석 결과의 구조화된 출력
- 저장형 인증 기반 provider 재사용
- 오류 발생 시 원인 확인 가능한 메시지와 종료 코드
- 단일 명령 실행 + 역할별 로그 확인 가능

### 이번 범위에서 제외
- 웹 대시보드
- 복잡한 정적 분석 엔진 내장
- 장기 워크플로우 오케스트레이션 서버화

## 운영 사용자 기준 실행 흐름

### 1. 초기 설정

```bash
node ./src/cli.js login --provider openai
node ./src/cli.js providers
```

### 2. 서비스 실행

```bash
multiverse-sec "보안이 강화된 login API 만들어줘"
```

### 3. 결과 확인
- 후보별 보안 이슈
- 방어 전략과 수정 제안
- 최종 선택 근거와 결과 코드

### 4. 실패 시 복구
- provider 상태 점검: `node ./src/cli.js providers`
- 기본 provider 전환: `node ./src/cli.js use --provider <name>`
- 역할별 재할당: `node ./src/cli.js assign --assign-role <role> --provider <name>`

## 품질 게이트
- [ ] 동일 입력에 대해 결과 형식이 깨지지 않는다.
- [ ] provider 실패/타임아웃 시 명확한 오류를 반환한다.
- [ ] 재실행 시 인증/설정이 정상 재사용된다.
- [ ] 테스트 스위트가 통과한다.
- [ ] 운영 문서 기준으로 신규 사용자가 10분 내 실행 가능하다.

## 최종 체크리스트
- [ ] 사용자 관점의 기본 시나리오(로그인 -> 실행 -> 결과 확인)가 완료된다.
- [ ] 실패 복구 시나리오(오류 -> 원인 확인 -> 재실행)가 동작한다.
- [ ] 실행 결과에 보안 검토 근거가 포함된다.
- [ ] 릴리스 가능한 수준의 안정성/운영성 검증이 완료된다.

## 결론
이 프로젝트의 핵심은 "보여주기용 연출"이 아니라 "반복 사용 가능한 서비스 품질"입니다. 생성, 공격, 방어, 합의의 흐름이 실제 운영 환경에서도 안정적으로 재현되어야 합니다.
